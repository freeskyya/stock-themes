Subtheme 01

單一企業案例：100 人公司導入 Agent 後的 CPU / Server 預算

假設一間 100 人的 SoC design house，平均每人約 3 個 agent workflow，估算因此新增的企業基礎設施需求。

最後更新：2026-04-04 18:57 CST

Internal Budget View
Executive Summary

核心摘要

對一間約 100 人的 SoC design house 來說，若平均每人導入約 3 個 agent workflow，企業通常不會先大規模購買 GPU 伺服器， 而是先補 CPU-heavy 的 infrastructure，用來支撐 orchestration、知識檢索、文件索引、log 分析、build / test 與內部流程自動化。 在雲端為主情境下，合理新增需求約為 2–4 台像樣的 server / workstation；若走 hybrid / 部分本地 AI，則會進一步增加 CPU、GPU 與 storage 投入。

3 個

300 個

約 30–60 個

Key Assumptions

分析假設

  • agent 主要協助研發、文件處理、內部檢索、bug / issue summary 與流程自動化。
  • 並非所有 agent 同時滿載，估算同時活躍率約 10%–20%。
  • 70% 活躍 agent 屬於輕量 orchestration / retrieval 類工作，20% 屬於中度分析，10% 才是較重任務。
  • 公司偏向先補 CPU / RAM / NVMe，再視資料敏感度與頻率決定是否加本地 AI 節點。
Key Takeaways

關鍵判斷

  • agent 化會先推升 CPU-heavy infrastructure 需求，而不是一開始就全面推爆 GPU 伺服器。
  • 雲端為主情境勝在低 CAPEX 與導入速度；Hybrid 情境勝在資料掌控與長期能力建設。
  • 對投資敘事來說，受惠的不只 GPU，還包括 server CPU、memory、SSD 與 enterprise infrastructure。

情境 A：大多數模型走雲端 API

目的： 以最低本地 infrastructure 成本，快速導入 agent workflow，先提升員工生產力與流程自動化程度。

方法： 把重模型推理交給外部雲端 API，本地只保留 agent orchestration、文件檢索、權限控管、workflow 排程與內部工具整合。

項目估算
Agent orchestration / docs / search 主機數量2–3 台（整體配置）
每台 CPU 規格16–32 cores / 64–128GB RAM / 1–4TB NVMe（每台）
每台節點預算約 US$4k–9k / 台
預算級別約 US$12k–28k（整體預算）
  • 上線快、維運簡單。
  • 適合先驗證 agent 導入 ROI。
  • 缺點是對外部 API 依賴高，敏感資料治理要更小心。

情境 B：Hybrid / 部分本地 AI

目的： 在維持 agent 生產力的同時，提高對敏感資料、內部知識庫與高頻工作流的掌控度，並為長期擴充建立內部 AI infrastructure。

方法： 除了保留 workflow / search / orchestration 本地能力外，額外部署 CPU-heavy nodes 與小型 GPU 節點，支援私有文件檢索、embedding、reranking、部分模型 inference 與低延遲 agent 任務。

項目估算

GPU node 預算約 US$8k–20k / 台
Storage / backup 預算約 US$4k–10k（整體預算）
預算級別約 US$30k–70k+（整體預算）
  • 較適合內部資料敏感、使用頻率高、希望降低長期外部依賴的公司。
  • 優點是掌控度更高、延遲更低、可客製化程度更好。
  • 缺點是前期 CAPEX 與維運複雜度都更高。

Investment Takeaway

對投資敘事的意義

  • 這類需求不是靠單一超大單，而是靠大量中小企業同步補 infrastructure。
  • 最先受惠的不是只有 GPU，還有 server CPU、memory、SSD、networking。
  • 對 AMD 來說，若企業 agent 化普及，EPYC / Ryzen Pro / workstation 類產品都可能受益。
Recommendation

一句結論

對 100 人公司來說，agent 化不是科幻，而是足以合理新增 2–4 台 CPU-heavy 機器的 IT 預算項目； 單家公司不誇張，但大量公司一起做，對 CPU 市場就很有感。

Model Assumptions

估算假設

  • 不是全球所有公司都會 agent 化，主要 relevant universe 為知識密集、可流程自動化的企業。
  • 若只算中重度受影響員工，平均每家導入企業約對應 40 人左右，因此 50 萬–150 萬家公司 roughly 對應 2,000 萬–6,000 萬人。
  • 小型公司以雲端為主，但仍可能補 1–3 台 CPU-heavy machine；中大型公司則逐步增加 hybrid / on-prem 比重。
  • 新增 infrastructure 不只支撐模型推理，也支撐 orchestration、retrieval、indexing、log analysis、workflow engine、storage 與內部控制層。
Key Interpretation

如何解讀

  • 這不是某家 hyperscaler 的單一大單，而是大量企業同步補 infrastructure 的 aggregate effect。
  • 短期 GPU 仍是 AI 主敘事，但 agent 化的企業內部 infrastructure 會讓 CPU / memory / storage 有更結構性的受惠邏輯。
  • 真正要看的是 adoption breadth，而不是單一 agent 功能本身有多炫。

核心摘要

如果企業 agent adoption 成為中長期趨勢，最直接受惠者不只會是 GPU，還包括 server CPU、memory、SSD / storage 與 enterprise infrastructure。對 AMD 而言，受惠點較分散，可同時連接 EPYC、Ryzen Pro / workstation、以及 AI 平台敘事；Intel 則更偏向企業 CPU / enterprise installed base；memory 供應鏈則會隨 server / edge / storage 一起被拉動。

AMD

AMD 受惠點

  • EPYC：企業 agent infrastructure 若擴張，server CPU 需求是最直接受惠點。
  • Ryzen Pro / workstation：中型公司若用 workstation / edge node 撐 agent workflow，AMD 也有戲。
  • AI 平台敘事：若 hybrid 部署增加，AMD 還能把 CPU + AI / GPU + platform 故事一起講。

特點：不是只吃單點，而是 CPU + 平台敘事一起受益。

Intel

Intel 受惠點

  • 企業 installed base 很深，agent adoption 若發生在既有企業 IT 內，Intel 會先吃到一部分 refresh。
  • CPU-only / conservative deployment 的公司，常更容易先沿著既有 Intel 環境擴。
  • 如果市場主敘事是企業 CPU refresh，而不是 pure AI GPU，Intel 也不會缺席。

特點：比較像既有 enterprise CPU 基盤的受惠者。

Memory / Storage

Memory / Storage 受惠點

  • 每新增一台 CPU-heavy node，通常都會拉高 DRAM / NVMe 配置。
  • agent workflow 很吃文件索引、log、cache、vector store，storage 是很實在的需求。
  • 這條線不靠單一贏家，屬於「系統性一起受惠」。

特點：屬於第二層但很穩的受惠鏈。

Bottom Line

一句結論

如果 agent 化變成企業標配，AMD 的優勢在於可以同時吃到 server CPU 與平台敘事；Intel 受惠邏輯則更偏向企業 CPU refresh；而 memory / storage 供應鏈則會因整體機器規格上升而得到更廣、更穩的第二層受惠。

Executive Summary

核心摘要

Infra Interpretation

為什麼不是只看總 agent 數

  • 很多 agent 是閒置的，像 app 安裝數，不等於實際同時活躍數。
  • 真正影響 compute / memory / storage 的，是同時運行中的 agent 與背後 workflow 深度。
  • 以 base case 來看，若同時在線 agent 約 8,000 萬個，企業與平台端就會開始持續補 CPU-heavy orchestration infrastructure。
1. 全球 Agent 數量 TAM

全球會有多少 agent？

  • base case 先抓活躍數位人口 40 億人、平均 2 個 agent / 人，則全球 agent 存量約 80 億個。
  • 真正 agent-native 的工作者、團隊與企業，平均 agent / 人可能明顯高於 2。
  • 因此長期來看，全球 agent identity / workflow 總量進一步往更高數字走，並不誇張。
2. 活躍 vs 同時在線

同時在線 agent 才是關鍵

  • 總存量像安裝基數；活躍量像 MAU / DAU；同時在線數則更接近 infra 壓力來源。
  • 即使 80 億 agent 存量中只有部分持續運作，也可能形成 8,000 萬級的同時在線規模。
  • 這是為什麼 orchestration、queue、search、memory、logging 層會先變重要。
3. Infrastructure Reverse-Map

對 CPU / memory / storage 的反推

  • CPU： 多數 agent workflow 的第一層需求不是大模型訓練，而是 orchestration、retrieval、ranking、tool execution、workflow control。
  • Memory： agent session、context window、cache、vector search、並發請求會一起拉高 RAM 需求。
  • Storage： 記憶、文件索引、log、artifact、history、backup 都會吃掉大量 storage。
  • 結論： 若全球進入 agent-native 社會，最先被拉動的不只 GPU，而是整個 CPU-heavy infrastructure stack。

Subtheme 06

哪些 Agent 需求會留在 edge？哪些會回到 server / cloud？

不是所有 agent 需求都會直接變成 server CPU。這個次主題把常見場景拆成 edge-first、hybrid、server-heavy 三類，幫你判斷需求會落在哪一層。

最後更新：2026-04-04 20:59 CST

Edge vs Server
Executive Summary

核心摘要

真正合理的 agent 基礎設施敘事，不是所有需求都推向 server / cloud，而是需求會重新分配到 edge、local hub、regional node 與 server 之間。凡是低延遲、高隱私、小模型可做、事件觸發型的任務，更容易留在 edge；凡是跨系統、多步驟、高併發、需要大型記憶與長期存檔的工作，則更容易回到 server / cloud。

提醒 / 輕健康 / IoT

最典型 hybrid

搜尋 / 個人知識流

最吃 server

企業 workflow / 大規模處理

Most Likely to Stay on Device

最可能被極小 edge agent 做掉的需求

  • 提醒 / 日程 / 生活管理：因為資料就在手機，邏輯簡單，延遲要求高。
  • 輕量健康監測：因為多數只是 anomaly flag，不需要大模型長篇推理。
  • 簡單安全監測 / IoT 控制：因為本地事件判斷後再上傳摘要就好，不需要全量上雲。
  • 訊息分類 / 輕摘要：很多是小型推理，適合 on-device inference。
Still Needs Server / Cloud

仍然會回到 server 的需求

  • 大型搜尋 / 跨平台資訊整合：牽涉大量索引、外部資料源與長鏈條檢索。
  • 多步驟企業 workflow：ERP / CRM / 文件 / code / ticket 串接，本質就是 server-side 協作問題。
  • 大規模內容處理：長影片、長音訊、文件批次整理、高併發客服，很難完全靠 edge 撐住。
  • 長期記憶與組織級 history：edge 可以存一部分，但大型記憶層與歷史資料還是偏 server。

Interpretation

怎麼看這三個情境？

  • 保守： server fleet 約 25 萬–40 萬台，比較像 base case，下限估法。
  • 中性： server fleet 約 40 萬–80 萬台，比較像 agent 普及後的正常擴張。
  • 積極： server fleet 約 80 萬–150 萬台，代表同時在線率更高、edge 分流沒那麼多、回 server 的高價值 workload 更多。
Existing Device Absorption

既有裝置吸收效應：為什麼個人 agent 需求不一定等於新增 PC 台數？

  • 全球每年本來就有大量筆電、手機、平板與穿戴裝置出貨，這些既有裝置會先吸收很大一部分個人 agent 需求。
  • 因此個人端更合理的敘事，不是「一個人多買一台 device」，而是裝置規格提升、換機週期加速、以及 edge compute 使用率上升。
  • 真正比較容易形成新增硬體需求的，通常不是前端 device unit，而是後端的 server、memory、storage 與 supporting infrastructure。

Phone-First

哪些需求最像手機端任務？

  • 提醒 / 行程 / 位置通知：因為高度即時、和人當下情境強相關。
  • 輕量健康監測：因為感測器資料就在手機與穿戴裝置端。
  • 訊息分類 / 輕摘要：很多是小任務、高頻、適合 on-device 先處理。
PC-First

哪些需求更像個人筆電任務？

  • 長文件處理、研究整理、報表與 coding assistant，往往在桌面工作流內發生。
  • 這類需求雖然可用 server 輔助，但主介面與主操作場景通常仍在筆電 / PC。
  • 因此 agent adoption 也可能更多反映成 PC 規格提升與換機節奏，而不只是 server 需求。

Investment Interpretation

對投資敘事的意義

  • 不能把所有 agent 需求直接換算成 server CPU 需求，因為一部分會被 edge 吃掉。
  • 但需求不是消失，而是分流成 edge compute、local storage、local memory 與 server infrastructure 共同成長。
  • 因此更合理的受惠鏈，不只是 server CPU，也包含手機 SoC、edge NPU、local SSD / NAND、以及後端 CPU / memory / storage。
